# Dynamic Island — Internationalization (English + Simplified Chinese)

**Date:** 2026-04-19
**Target:** GNOME Shell 50 extension `dynamic-island@hacksynth.github.io`
**Status:** Approved for planning

## 1. Goal

Localize every user-visible string in the extension in two languages — English (source) and Simplified Chinese (`zh_CN`) — using the standard GNOME Shell 45+ gettext pipeline. Language follows the system locale; no user-facing language switcher.

The pipeline is built so adding a third language later costs only a new `po/<lang>.po` file plus one translator pass — no code changes.

## 2. Scope

### In scope
- All UI chrome visible at runtime: flash labels, accessibility names/descriptions, right-click menu, `Main.notify` messages.
- The entire `Adw.PreferencesWindow` (group titles, row titles, dropdown items, custom-text entry).
- `gschema.xml` `<summary>` and `<description>` for every key (surfaced by `dconf-editor` and `gsettings describe`).
- Provider display names shown in prefs and in the "Disable X" menu item.

### Explicitly NOT translated
- `providerId` constants (`"media"`, `"keyboard"`, etc.) — they are stable identifiers used in gschema keys and activity ids.
- MPRIS track titles / artist names and notification bodies — content is controlled by the source app.
- Input-source identifiers (`Layout · zh`) — the trailing token is the keyboard layout id, not a word.
- Any `console.log`/`console.error` text — developer-facing only.

### Explicitly out of scope (v1)
- Any language other than `en` and `zh_CN`.
- A per-extension language override setting. System `LANG` / `LC_MESSAGES` is the only selector.
- Runtime language switching (user must log out and back in to change, same as all GNOME apps).

## 3. Approach

GNOME Shell 45+ ships a built-in gettext integration in both the extension runtime and the preferences process:

- `resource:///org/gnome/shell/extensions/extension.js` exports `gettext`, `ngettext`, `pgettext` — used by runtime modules (shell process).
- `resource:///org/gnome/Shell/Extensions/js/extensions/prefs.js` exports the same symbols — used by `prefs.js` (separate prefs process).

Both look up `metadata.json`'s `"gettext-domain"`, bind it to `<extension-dir>/locale` automatically on load, and fall back to `/usr/share/locale`. We don't call `bindtextdomain` ourselves.

Alternatives considered and rejected:
- **Home-grown JSON table + manual lookup function.** Reinvents `.po` tooling (`msgmerge`, plural forms, gschema `.its`-based extraction). No upside.
- **`libintl` direct bind via GJS.** Skips the base-class auto-init, so we'd have to replicate it in two processes. No upside.

## 4. Architecture

```
dynamic-island@hacksynth.github.io/
├── metadata.json                    # + "gettext-domain": "dynamic-island"
├── schemas/
│   └── org.gnome.shell.extensions.dynamic-island.gschema.xml
│                                    # + gettext-domain="dynamic-island" attr on <schemalist>
├── locale/
│   └── zh_CN/LC_MESSAGES/
│       └── dynamic-island.mo        # generated by scripts/compile-mo.sh
├── po/
│   ├── POTFILES.in                  # list of files xgettext scans
│   ├── dynamic-island.pot           # template, regenerated by scripts/update-po.sh
│   └── zh_CN.po                     # hand-maintained translation
├── extension.js                     # import { gettext as _ } from extension base
├── prefs.js                         # import { gettext as _ } from prefs base
└── src/
    ├── provider-display.js          # NEW — providerId → display name map
    └── … (all existing modules gain `_()` wraps on user-visible strings)
```

### 4.1 Two processes, two import sources

Shell-process modules (`extension.js`, `src/**`) import:
```js
import {gettext as _, ngettext} from 'resource:///org/gnome/shell/extensions/extension.js';
```

Prefs-process module (`prefs.js`) imports:
```js
import {gettext as _, ngettext} from 'resource:///org/gnome/Shell/Extensions/js/extensions/prefs.js';
```

Both resolve to the same `dynamic-island` domain because `metadata.json` declares it. A translation added to `zh_CN.po` appears in both surfaces after recompiling `.mo`.

### 4.2 `src/provider-display.js` (new module)

Currently `prefs.js` uses raw `providerId` strings ("volume-brightness", "media") as UI titles, and `interaction-controller.js` does `Disable ${providerId}`. Both are ugly in English and illegible in Chinese. Centralize the display name so both callers translate the same msgid:

```js
// No gettext import here — the module has to work in both the shell and prefs
// processes, so the caller passes its own _ in. N_ is the conventional no-op
// marker that tells xgettext "this string needs translating but we'll translate
// it lazily at call time". xgettext is configured with --keyword=N_.
const N_ = s => s;

const NAMES = {
    media: N_('Media'),
    notification: N_('Notifications'),
    'volume-brightness': N_('Volume / Brightness'),
    power: N_('Power'),
    keyboard: N_('Keyboard'),
};

export function providerDisplayName(id, gettext) {
    return gettext(NAMES[id] ?? id);
}
```

The `N_(…)` markers cause `xgettext` to emit msgids; the caller's `gettext(…)` does the actual lookup at runtime with the right per-process domain binding.

### 4.3 Placeholder convention

All format strings use `%s`, `%d`, `%d%%` — never JS template literals — so `xgettext` can extract them:

```js
// BAD: _(`Caps Lock ${on ? 'on' : 'off'}`)   ← xgettext sees one opaque string
// BAD: `${_('Battery low')} ${pct}%`          ← translator can't reorder

// GOOD:
on ? _('Caps Lock on') : _('Caps Lock off')
_('Battery low — %d%%').format(pct)
ngettext('%d new', '%d new', n).format(n)
```

A 4-line `src/i18n.js` supplies `String.prototype.format` via a `format(str, ...args)` helper (no prototype pollution):

```js
export function format(str, ...args) {
    let i = 0;
    return str.replace(/%[sd]/g, () => String(args[i++]))
              .replace(/%%/g, '%');
}
```

Callers: `format(_('Battery low — %d%%'), pct)`.

## 6. Build & install pipeline

Two shell scripts. No Node build chain, no Makefile.

### 6.1 `scripts/update-po.sh` (maintainer)

Run after adding or changing any translatable string. Produces updated `po/dynamic-island.pot` and `po/*.po` (both committed to git).

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."
EXT=dynamic-island@hacksynth.github.io

xgettext --from-code=UTF-8 --language=JavaScript \
    --keyword=_ --keyword=N_ --keyword=ngettext:1,2 \
    --package-name=dynamic-island \
    --output=po/dynamic-island.pot \
    $(cat po/POTFILES.in)

xgettext --its=/usr/share/gettext/its/gschema.its \
    --join-existing --output=po/dynamic-island.pot \
    "$EXT"/schemas/*.xml

for po in po/*.po; do
    msgmerge --update --backup=none "$po" po/dynamic-island.pot
done
```

### 6.2 `scripts/compile-mo.sh` (run on install)

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."
EXT=dynamic-island@hacksynth.github.io

for po in po/*.po; do
    lang=$(basename "$po" .po)
    dir="$EXT/locale/$lang/LC_MESSAGES"
    mkdir -p "$dir"
    # --check-format fails the build if translator dropped a %s / %d
    msgfmt --check-format "$po" -o "$dir/dynamic-island.mo"
done
```

### 6.3 `scripts/install-dev.sh` change

Insert one line before the `glib-compile-schemas` step:
```bash
bash "$(dirname "$0")/compile-mo.sh"
```

### 6.4 `package.json`

## 7. Testing

`tests/i18n.test.js` — runs under `node --test`, no shell dependency:

- Assert `po/dynamic-island.pot` exists and is non-empty.
- Assert `po/zh_CN.po` exists and every non-header entry has a non-empty `msgstr`. Parse with a minimal regex: an entry is a run of `msgid "…"` / `msgid_plural "…"` / `msgstr "…"` / `msgstr[n] "…"` lines. Skip the header (`msgid ""` + `msgstr ""` at the top).
- Assert `po/dynamic-island.pot` contains every msgid we expect (spot-check a stable subset: `Caps Lock on`, `Charger connected`, `Behavior`, `Disable %s`). This catches accidental deletions or `xgettext` misconfigurations.

Runtime translation is verified by hand, captured in `scripts/verify-checklist.md`:

- `LANG=en_US.UTF-8 gnome-extensions prefs dynamic-island@hacksynth.github.io` — English surfaces.
- `LANG=zh_CN.UTF-8 gnome-extensions prefs dynamic-island@hacksynth.github.io` — Chinese surfaces.
- Toggle Caps Lock, plug/unplug power, adjust volume under both locales; verify the flash labels change.
- `gsettings describe org.gnome.shell.extensions.dynamic-island transient-duration-ms` under both locales — the summary changes language.

## 8. Risks and mitigations

1. **`/usr/share/gettext/its/gschema.its` path may vary between distros.** Mitigation: `update-po.sh` checks for the file and falls back to `locate gschema.its` with a clear error message if missing.
2. **`xgettext` may not recognize JS template literals.** Mitigation: the placeholder convention (§4.3) forbids them for translated strings; enforced by the test that scans the `.pot` for known msgids.
3. **Translators may forget `%s` / `%d` in the Chinese translation.** Mitigation: `msgfmt` is strict (`--check-format`); run it in `compile-mo.sh` and fail the install if placeholders don't match.
4. **Adding a string without re-running `update-po.sh`.** Mitigation: a pre-commit nicety later, but v1 relies on code review + the `tests/i18n.test.js` spot-check failing.

## 9. Success criteria

- `LANG=zh_CN.UTF-8 gnome-extensions prefs …` shows a fully Chinese preferences window.
- Caps Lock / charger / notification flashes appear in Chinese under `LANG=zh_CN.UTF-8` and English under `LANG=en_US.UTF-8`.
- `tests/i18n.test.js` passes, covering: `.pot` exists, `zh_CN.po` has no empty `msgstr`, expected msgids present.
- `npm run i18n:update` after adding a new English string produces a clean diff on `po/dynamic-island.pot` and `po/zh_CN.po` (with the new entry as an empty `msgstr` awaiting translation).
- Adding a third language is a single new file (`po/<lang>.po`) — no code change.
